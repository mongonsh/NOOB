# Real-Time Voice Streaming Implementation Guide

## Overview

This document describes the real-time voice streaming implementation for the Emergency Voice Assistant feature. The system uses ElevenLabs for text-to-speech generation with streaming support, enabling low-latency audio responses.

## Architecture

### Backend Components

1. **ElevenLabs Service** (`services/elevenlabs_service.py`)
   - Provides text-to-speech streaming via `text_to_speech_stream()` method
   - Yields audio chunks as they are generated by ElevenLabs API
   - Supports multiple voice models (Rachel, Adam, Bella, Josh)

2. **Emergency Assistant Service** (`services/emergency_assistant_service.py`)
   - Implements `generate_response_stream()` method for streaming responses
   - Coordinates text generation and audio streaming
   - Emits events via Socket.IO for real-time communication

3. **Socket.IO Handlers** (`app.py`)
   - Handles `emergency:message` event with streaming support
   - Emits `emergency:audioStream` events for each audio chunk
   - Emits `emergency:audioStreamComplete` when streaming finishes

### Frontend Components

1. **EmergencyScene** (`frontend/src/scenes/EmergencyScene.js`)
   - Listens for `emergency:audioStream` events
   - Uses Web Audio API for low-latency playback
   - Buffers and plays audio chunks seamlessly
   - Falls back to HTML5 Audio element if Web Audio API fails

## How It Works

### Backend Flow

1. User sends a message via Socket.IO
2. Emergency service generates text response
3. Text is sent to ElevenLabs for streaming TTS
4. Audio chunks are received from ElevenLabs (1024 bytes each)
5. Each chunk is base64-encoded and emitted to client
6. Completion signal is sent when all chunks are delivered

```python
# Example: Streaming response generation
for chunk_data in service.generate_response_stream(session_id, message):
    if chunk_data['type'] == 'audio_chunk':
        # Encode and emit chunk
        audio_b64 = base64.b64encode(chunk_data['chunk']).decode('utf-8')
        emit('emergency:audioStream', {
            'sessionId': session_id,
            'audioChunk': audio_b64,
            'chunkNumber': chunk_data['chunk_number']
        })
```

### Frontend Flow

1. Client receives `emergency:audioStream` event
2. Audio chunk is decoded from base64
3. Web Audio API decodes MP3 data
4. Audio buffer is scheduled for playback
5. Chunks are played seamlessly with calculated timing
6. Stream is finalized when `emergency:audioStreamComplete` is received

```javascript
// Example: Handling audio stream chunk
_handleAudioStreamChunk(audioChunkB64, chunkNumber) {
    // Decode base64 to binary
    const binaryString = atob(audioChunkB64);
    const bytes = new Uint8Array(binaryString.length);
    
    // Decode and play with Web Audio API
    this.audioStreamContext.audioContext.decodeAudioData(
        bytes.buffer,
        (audioBuffer) => {
            const source = this.audioStreamContext.audioContext.createBufferSource();
            source.buffer = audioBuffer;
            source.connect(this.audioStreamContext.audioContext.destination);
            source.start(startTime);
        }
    );
}
```

## Performance Characteristics

### Latency

- **First chunk latency**: ~200-500ms (time to first audio playback)
- **Chunk size**: 1024 bytes (optimal for streaming)
- **Total chunks**: Varies by response length (37-132 chunks in tests)
- **Playback**: Seamless with Web Audio API scheduling

### Bandwidth

- **Audio format**: MP3 (compressed)
- **Bitrate**: ~128 kbps (ElevenLabs default)
- **Chunk transmission**: Base64-encoded (33% overhead)
- **Total data**: ~37-135 KB for typical responses

## Error Handling

### Backend Errors

- **ElevenLabs API failure**: Falls back to non-streaming mode
- **Session not found**: Returns error to client
- **Streaming interruption**: Emits error event to client

### Frontend Errors

- **Web Audio API failure**: Falls back to HTML5 Audio element
- **Decode error**: Logs error and continues with next chunk
- **Connection loss**: Cleans up audio stream and shows error

## Usage Example

### Sending a Message with Streaming

```javascript
// Frontend: Send message with streaming enabled
this.socket.emit('emergency:message', {
    sessionId: this.sessionId,
    message: 'What should I do in an emergency?',
    streamAudio: true  // Enable streaming
});

// Listen for streaming events
this.socket.on('emergency:audioStream', (data) => {
    this._handleAudioStreamChunk(data.audioChunk, data.chunkNumber);
});

this.socket.on('emergency:audioStreamComplete', (data) => {
    console.log('Streaming complete:', data.totalChunks, 'chunks');
});
```

### Backend: Generating Streaming Response

```python
# Generate streaming response
for chunk_data in service.generate_response_stream(session_id, message):
    if chunk_data['type'] == 'text':
        # Send text response
        emit('emergency:transcript', {...})
    elif chunk_data['type'] == 'audio_chunk':
        # Stream audio chunk
        emit('emergency:audioStream', {...})
    elif chunk_data['type'] == 'audio_complete':
        # Signal completion
        emit('emergency:audioStreamComplete', {...})
```

## Troubleshooting

### No Audio Playback

1. Check browser console for Web Audio API errors
2. Verify ElevenLabs API key is set in `.env`
3. Check network tab for Socket.IO events
4. Ensure audio permissions are granted

### Choppy Audio

1. Check network latency
2. Verify chunk scheduling logic
3. Monitor browser performance
4. Consider fallback to buffered playback

### High Latency

1. Check ElevenLabs API response time
2. Verify Socket.IO connection quality
3. Monitor server CPU usage
4. Consider using CDN for audio delivery
